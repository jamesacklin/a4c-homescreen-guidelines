Overview

These are design specifications for several independently-swappable components for the Apps4Clubz iOS client application homescreen. This is a living document and these specifications are open to review.

The goals of these designs are to (1) provide club administrators with the means to visually distinguish their club's app from others, and to (2) provide club members with a meaningful and useful starting experience from the home screen of their club's application.

These element specifications provide a “mix and match” approach between various configurations of module navigation, welcome message, background image, and content presentation configurations.

Backgrounds

Six different background image treatments.

Admins should be able to upload their own background image for the homescreen. The shortest side of the user-uploaded image should be scaled to fit within each masked area, and then cropped from the center. The background photo should also be set to 80% opacity, with a pure black (rgb(0,0,0)) background to enhance contrast with overlaid text.

There are plenty of suitable, free-to-use background images available on Unsplash.

Welcome Messages

Three different welcome banner schemes.

1. User-Personalized Welcome Message

Use the club's logo in a circle crop, the user's first name, and most recent login together to create a personalized welcome message. This requires very little customization on the part of the club admin, and does not offer any way to alter the message members see in this section of the homescreen.

On first login, the message should read "This is your first login." Subsequent sessions should list the last login in relative terms. If the last login is within 24 hours, the last login text should read, "Your last login was earlier today."

The welcome message should make use of as much horizontal space as possible. If the user's name is too long to fit on the same line as the string "Welcome, ", the user's name should wrap to the next line.

2. Customizable Welcome Message

Use the club's logo in a circle crop, and the club admin should be able to provide 100 characters of text to place into the banner. This gives the club admin an opportunity to customize the message members see in this section of the homescreen.

The text should wrap naturally. The banner's size should grow or shrink with the number of lines of text, and hence shift all elements below it up or down.

3. Custom Header Image

With this option, club admins should be able to upload their own 700x380 image to use as the welcome message. This gives club admins complete creative control over this section of the homescreen.

Navigation Styles

Four different styles for navigation content.

1. Bottom Tabs

Standard iOS tabbed navigation, which has already been implemented.

2. Top Tabs

Similar to the already-implemented bottom tab navigation, except flipped to appear on the top of the screen.

The rest of the app content—on all screens—will need to shift down to fit the navigation bar on top.

3. Tiles

Tile navigations are a crisp way to display navigation options and give end-to-end coverage, eliminating the need for an overflow menu and creating a very full-bodied effect for the user. This navigation pattern would be optimal for clubs with a low module count.

The user can scroll the stack of tiles back and forth horizontally to see all the options. This pattern appears frequently in Apple apps, such as the App Store itself and Apple Music.

The tiles should be spaced 8px apart with a slight drop shadow. The first and last should have a slightly larger buffer on their left and right sides, respectively. The third tile in the series should intentionally run off the side of the screen—this is a common visual trick Apple apps use to show that there is more content hidden, to be shown when the user drags the row left to scroll.

Note: this navigation pattern introduces a "hub-and-spokes" approach to navigation. If the user navigates away from the home screen using one of these tiles, they must tap "back" enough times to get back to the home screen to navigate to a different module. This can be good or bad: if the club's module count is low, then there is a less-likely chance that they will need to switch between modules quickly.

The rest of the homepage content will need to shift around to fit the tiles in the vertical space needed. Furthermore, their vertical position will depend on the banner message chosen.

4. Floating Action Button

Adapted from Google's Material Design toolkit, the Floating Action Button is an unobtrusive, persistent button in the lower-right of the user's screen. The button should stay on the screen at all times with a pinned location, "floating" above the content that changes beneath it.

When the user taps the Floating Action Button, the button should open a label-free menu similar to the bottom tab bar, using an animated transition. Look to Google's transition guidelines for guidance. The overflow menu should open the same side-panel navigation we have now. Tapping the screen outside the bounds of the menu should close the menu and transition the shape back into a button.

Content Tiles

Top-level homescreen information display for each module.

Standard Module Contents

Regardless of style, each module should have an associated template for homescreen tiles. The general form is headline, content (three lines), and Call-to-Action button (20-character limit, with defaults). There are some exceptions to this: the Loyalty module should display the user's loyalty point balance and barcode, the Course Conditions and Events tiles break the pertainent information out into table views of their own.

The Call to Action button, when tapped, should navigate the user to the pertainent item in the corresponding Module. For example, if the user taps "RSVP Now" on an Event tile, they should be taken to the appropriate Event.

As a suggestion, the homescreen tile content should be separate fields when an administrator is creating a new item within a Module.
